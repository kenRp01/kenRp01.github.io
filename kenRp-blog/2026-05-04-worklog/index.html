2026-05-04の作業ログ: worklog-blogger拡張とkenRp-blog本番化

2026年05月04日
#codex#worklog

2026年05月04日は、日次作業ログを生成する worklog-blogger スキルの出力品質を上げる改修と、kenRp-blog を実運用できる形に整える作業が中心だった。単なる要約ではなく、日次ワークログと週次ナレッジ記事を分離し、記事構造・公開契約・デプロイ経路まで一貫して整理した。

作業ログ

1. worklog-blogger を日次ログと週次記事の2系統に再設計した

利用した技術

  • Python 3: collect_worklog.py の集約処理、frontmatter生成、出力先切り替えに利用した。
  • unittest: 出力構造の回帰を防ぐため、test_collect_worklog.py を更新した。
  • Codex session logs: 作業スレッド名、実行コマンド、判断理由の抽出元として利用した。
  • Git history: その日に実際に行われた改修単位をコミット列から復元するために利用した。

アーキテクチャ

  • 入力層: Codex のセッションログとローカル Git 履歴を読む。
  • 集約層: collect_worklog.py が対象日を基準にセッションとコミットを束ね、日次ワークログ用のデータと週次ナレッジ記事用のデータを分岐して組み立てる。
  • 表現層: kenRp-blog の frontmatter 契約に合わせて Markdown を生成する。
  • 品質ゲート: --dry-run で内容確認し、問題がなければ --write でファイルを保存する。

この構成にしたことで、同じ一次情報から「作業再開のための記録」と「公開向けの知見記事」を別目的の成果物として出し分けられるようになった。

選定理由

  • 日次ログと週次記事を分けた理由: 日次ログは手戻りなく作業を再開するための記録が主目的で、週次記事は他者が再利用できる学びの整理が主目的だから。
  • Python の既存スクリプトを拡張した理由: 既にセッションログと Git をまたいだ収集処理があり、新規ツールを増やすより既存資産を育てる方が保守しやすかったため。
  • unittest を使った理由: セクション追加や文面変更で出力契約を崩しやすく、文字列ベースでも最低限の回帰検知が必要だったため。

実施手順

  1. 既存の worklog-blogger の責務を確認し、日次ワークログと週次ナレッジ記事を別モードで出力する方針を固めた。
  2. collect_worklog.py を更新し、worklog と weekly-blog の2モードを切り替えられるようにした。
  3. 日次ワークログには、各タスクごとに「利用した技術」「アーキテクチャ」「選定理由」「実施手順」「作業時の課題」「進捗状況」「次に検討した方が良いこと」を必須で出すよう調整した。
  4. 週次記事は導入・採用判断を中心に説明する構造へ寄せ、日次ログの焼き直しにならないよう見出しを分離した。
  5. 公開前の確認事項を本文から外し、品質ゲートとして扱うよう整理した。
  6. セッションログからコマンドと判断メモを拾い、ワークログに具体性を持たせる改修を追加した。
  7. 抽出したコマンドは Markdown の bash コードブロックで表示するように変更し、本文の可読性を上げた。
  8. 変更に合わせてテストを更新し、出力契約が崩れていないことを確認できる状態にした。

sed -n '1,240p' ~/.codex/skills/worklog-blogger/scripts/collect_worklog.py

重要な改修箇所

  • skills/worklog-blogger/scripts/collect_worklog.py 日次・週次の出力分岐、見出し構造、セッション由来のコマンド抽出、判断メモ抽出、bash コードブロック化を実装した中心ファイル。
  • skills/worklog-blogger/scripts/test_collect_worklog.py 追加した必須見出しや出力形式に追従するテストを整備した。
  • skills/worklog-blogger/SKILL.md 生成対象、品質ゲート、各成果物の目的と必須見出しを明文化した。

技術選定・判断メモ

  • まず dry-run を必須にしたのは、セッションログ由来の文面はそのままだと汎用的になりすぎたり、逆に内部事情が出すぎたりするため。
  • 公開前の確認事項を本文に残さず品質ゲートへ寄せたのは、公開記事として読んだときのノイズを減らしつつ、運用上の確認事項は維持したかったため。
  • コマンドを箇条書きではなく bash コードブロックにしたのは、再現手順としての読みやすさを優先したため。

作業時の課題

  • セッションログだけでは、実作業の中身が抽象的にしか残っていないケースがあり、Git 履歴を併用しないと作業粒度が粗くなる。
  • 自動生成文は frontmatter や見出しは整っていても、公開向けの自然な文章としては薄くなりやすい。
  • コマンド抽出を増やすほど、ローカル専用パスや内部的な操作が混ざるリスクが上がるため、公開可能な形への正規化が必要だった。

進捗状況

  • 日次ワークログと週次ナレッジ記事を別ファイルで扱う基本構造は実装できた。
  • ワークログに技術背景、アーキテクチャ、判断理由、コマンドを含める形まで到達した。
  • 自動生成の精度は上がったが、公開前に人手で文面を整える前提はまだ残っている。

次に検討した方が良いこと

  • セッションログの抽出粒度を上げ、どのリポジトリのどの変更に紐づく作業かを自動で結び付けられるようにしたい。
  • コミット群をタスク単位に自動クラスタリングできると、日次ログの章立てがさらに安定する。
  • 秘匿情報やローカル固有値の検出ルールは、正規表現だけでなく禁止語辞書も併用した方が安全。

2. kenRp-blog の記事契約と本番公開フローを整備した

利用した技術

  • Gatsby: ブログの静的サイト生成基盤として利用した。
  • GitHub Pages / GitHub Actions: kenRp-blog のビルド成果物を公開する経路として利用した。
  • Netlify CMS config: Markdown 記事の frontmatter 契約と管理画面項目の整合に利用した。
  • Playwright: 画面上の主要導線やフォーカス状態の確認用 E2E テストに利用した。

アーキテクチャ

  • コンテンツ層: _data/blog/*.md を記事ソースとし、frontmatter で公開対象かどうかを制御する。
  • ビルド層: Gatsby が Markdown とテンプレートを結び付け、一覧・詳細・タグ・カテゴリページを生成する。
  • 配信層: GitHub Actions で npm ci、npm test、npm run build を通して静的成果物を GitHub Pages へ配置する。
  • 運用層: Markdown 中心で記事を管理しつつ、CMS 設定も新しい frontmatter に追従させる。

この構成により、記事の作成はローカル Markdown ベースのまま維持しつつ、公開面では Gatsby と Pages に責務を分離できた。

選定理由

  • Gatsby を継続利用した理由: 既存のブログ資産があり、全面移行よりも既存構成を本番利用可能に寄せる方が変更コストが低かったため。
  • GitHub Pages を採用した理由: 静的配信との相性が良く、公開先 URL が明確で個人ブログ用途に十分だったため。
  • draft frontmatter を正式化した理由: サンプル記事や下書きを残しつつ、公開対象だけを一覧・詳細生成へ流せるようにするため。

実施手順

  1. kenRp-blog を個人技術ブログとして使う前提を整理し、サンプル記事を消すのではなく draft 扱いにする方針を決めた。
  2. 記事 frontmatter に draft、tags、category、metaDescription を含める契約へ寄せた。
  3. gatsby-node.js と一覧クエリ側を調整し、draft: true の記事を公開ページ生成から外す流れを作った。
  4. トップページ、記事カード、記事詳細、About などを実運用向けに整え、プロフィール導線や記事一覧導線を追加した。
  5. site-meta-data.json や static/admin/config.yml を更新し、サイト名・説明・公開 URL・記事項目を実態に合わせた。
  6. GitHub Actions の Pages workflow を追加し、ビルド後の public を GitHub Pages に公開できる形へ整理した。
  7. 生成物を kenRp01.github.io 側へ反映し、公開済みサイトの導線修正まで行った。

実際に実行したコマンド

git -C ~/git/kenRp-blog show --name-only --format=medium b81715a
git -C ~/git/kenRp01.github.io show --name-only --format=medium f176c5d d532a72
rg --files kenRp-blog/_data/blog | tail -n 5

重要な改修箇所

  • gatsby-config.js, gatsby-node.js pathPrefix、記事公開条件、ビルド対象の制御に関わる中心設定。
  • src/pages/index.js, src/templates/blogTemplate.js, src/components/post-link.js 記事一覧と詳細画面の導線、メタ情報表示、視認性の改善に関わる UI 層。
  • .github/workflows/pages.yml 静的ビルドから GitHub Pages 配信までの自動化を担う workflow。
  • scripts/validate-content.js frontmatter 契約の検証と、公開記事の最低限の整合確認を担う補助スクリプト。

技術選定・判断メモ

  • 旧テンプレート由来のサンプル記事を即削除しなかったのは、ビルドの比較対象と初期データとして残しつつ公開面だけ制御したかったため。
  • Markdown 中心運用を維持したのは、日次ワークログ自動生成との相性が良く、Git 管理しやすかったため。
  • 本番デプロイを workflow に寄せたのは、手元ビルド依存を減らし、公開再現性を上げるため。

作業時の課題

  • 既存テーマ由来の構成がサンプルブログ前提だったため、個人技術ブログとして必要な契約に合わせて複数箇所を同時に直す必要があった。
  • Pages 向けの URL、pathPrefix、生成物配置先がずれると公開後にリンク切れが起きるため、設定の整合確認が重要だった。
  • 記事 frontmatter の項目追加は、生成側、一覧取得側、CMS 設定側を同時に更新しないと片手落ちになる。

進捗状況

  • kenRp-blog は本番運用を前提にした記事契約と公開フローまで一通り整備できた。
  • GitHub Pages への配信物更新と公開導線の修正まで完了している。
  • 今後は記事を増やすフェーズへ進める状態になった。

次に検討した方が良いこと

  • worklog-blogger が生成する日次ログと、kenRp-blog 側の frontmatter 契約をより厳密に同期させたい。
  • 公開記事用のサムネイルやカテゴリ設計を増やし、記事一覧の情報密度を上げたい。
  • 日次ワークログから週次ナレッジ記事を自動生成する運用を、実際の公開サイクルにどう組み込むか決める必要がある。
