静的ブログと共有アプリの運用を整えた一週間

2026年05月15日
#codex#automation#knowledge

今週は、個人技術ブログ kenRp-blog の公開運用、共有買い物リストアプリ shared-shopping-app の認証・共有設計、Go製APIサンプル go-sample-api の機密情報対策を進めた。

テーマはばらばらに見えるが、共通しているのは「小さく作ったものを、実際に使える状態へ寄せる」ことだった。画面を整えるだけでは足りず、問い合わせ、認証、権限、秘密情報、公開手順まで含めて見直す必要がある。

行ったこと

kenRp-blog では、公開記事のサムネイルを記事内容に合わせ、worklog系の下書きを公開一覧から外した。さらに Contact ページを改善し、GitHub Issues で公開相談できる導線と、公開しづらい内容を送る問い合わせフォームを分けた。

shared-shopping-app では、ゲスト利用中心だった買い物リストを、メール認証済みユーザーでも使える共有機能へ寄せた。Next.js の Route Handler を追加し、リスト、商品、共有メンバーの操作を Supabase 上の認証ユーザーと結びつけた。

go-sample-api では、コミット済みだった .env を除去し、.env.example、.gitignore、Docker Compose、README、セキュリティテストをまとめて見直した。履歴に残った秘密情報も検査し、再発防止のために gitleaks とテストを併用した。

過程

最初に整えたのは kenRp-blog の公開面だった。Markdownで記事を管理する静的ブログでは、frontmatterの draft、thumbnail、category、tags が一覧や詳細ページへ直結する。worklogを公開一覧から外し、読者向けの記事だけが並ぶようにしたことで、サイトの役割がはっきりした。

次に問い合わせ導線を見直した。最初は GitHub Issues へのリンクだけだったが、すべての問い合わせを公開Issueにするのは使いにくい。そこで Contact ページにフォームを追加し、後から mailto: ではなく FormSubmit のAjax送信に変更した。送信後に外部画面へ飛ばず、ページ内で完了状態を返す形にしたことで、静的サイトのまま問い合わせ体験を改善できた。

並行して、shared-shopping-app では共有機能の土台を作り直した。ローカル保存だけならIndexedDBで十分だが、複数人で同じリストを見るには、誰が所有者で、誰が共有メンバーで、誰が編集できるのかをサーバー側で判断する必要がある。そこで Supabase Auth のユーザーを基準に、Route Handler 経由でリストと商品を読み書きする構成へ寄せた。

最後に、go-sample-api の .env 問題を整理した。秘密情報は現在のファイルから消すだけでは不十分で、Git履歴にも残っていないか確認する必要がある。.env.example で必要なキーだけを示し、実値は追跡しない構成にしたうえで、gitleaks で履歴と作業ツリーを検査した。

利用した技術

kenRp-blog では Gatsby 2、React、Markdown frontmatter、Sass、Playwright を使った。記事データは _data/blog/*.md に置き、draft: false の記事だけを一覧・詳細ページに生成する。問い合わせフォームはReact stateで送信状態を管理し、FormSubmit のAjaxエンドポイントへ fetch する。

shared-shopping-app では Next.js 15、React 19、TypeScript、Supabase Auth、Supabase Database、IndexedDB、Zod、Playwright を使った。UIから直接DBを触らず、クライアントのデータアクセス層とNext.js APIを経由して共有リストを扱う。

go-sample-api では Go、Echo、GORM、PostgreSQL、Docker Compose、Goのテスト、gitleaks を使った。アプリ本体の機能追加ではなく、設定ファイル、開発環境、セキュリティ検査の整備が中心だった。

アーキテクチャ

kenRp-blog は、ソースリポジトリと配信リポジトリを分ける構成で運用している。ソース側でMarkdown、React、Sassを編集し、ビルド済みのHTML、JS、CSS、画像を kenRp01.github.io 側へ反映する。静的サイトなのでサーバー処理は持たず、問い合わせ送信は外部フォームサービスへ委ねる。

shared-shopping-app は、画面層、データアクセス層、API層、Supabase層に分けた。画面は AuthForm やリスト詳細コンポーネントが担当し、データ取得は local-store 経由に集約する。API層ではBearerトークンからユーザーを特定し、所有者または共有メンバーだけが編集できるようにする。

go-sample-api は、アプリケーションコードとは別に設定管理の層を整えた。実値を持つ .env はローカルだけに置き、リポジトリには .env.example を残す。Docker Compose とREADMEもこの前提に合わせ、テストとスキャンで漏えいを検出する。

選定理由

kenRp-blog の問い合わせでAjax送信を選んだ理由は、静的サイトの構成を保ったまま、利用者を外部ページへ飛ばさずに送信体験を完結させたかったからだ。自前APIを追加すると運用範囲が広がるため、まずは無料で使える外部フォームサービスに寄せた。

shared-shopping-app で Supabase を中心にした理由は、共有リストでは認証と永続化を同じ基盤で扱える方が設計しやすいためである。IndexedDB はゲスト利用には便利だが、複数ユーザーの共有状態を正として管理するには向いていない。

go-sample-api で .env.example と gitleaks を併用した理由は、人間の注意だけでは秘密情報の混入を防ぎきれないためである。テンプレートで使い方を示し、テストとスキャナで検出するほうが再発しにくい。

導入手順

まず公開面を整える。記事のfrontmatterを見直し、公開する記事は draft: false、下書きや作業ログは draft: true に分ける。記事タイトルに合うサムネイルを設定し、一覧ページで読者が内容を判断しやすくする。

次に問い合わせ導線を追加する。公開相談用にGitHub Issuesを残し、非公開寄りの連絡用にフォームを置く。静的サイトで運用する場合は、フォーム送信後の遷移や失敗時の表示を明示し、Playwrightで最低限の回帰確認を入れる。

共有機能では、先に権限の境界を決める。所有者、共有メンバー、未参加ユーザー、公開リンク閲覧者を分け、編集可能な操作をAPI側で判定する。UIはそのAPIを呼ぶだけにして、権限判定をクライアントへ寄せすぎない。

秘密情報対策では、.env を追跡対象から外し、.env.example だけを残す。すでに公開履歴へ入っていた場合は、HEADの修正だけで終わらせず、履歴検査、必要なら履歴修正、値のローテーションまで考える。

導入時の課題

静的サイトは運用が軽い一方で、問い合わせや認証のような動的処理をどこに置くかの判断が必要になる。今回はContactフォームを外部サービスに寄せたが、問い合わせ件数やスパムが増えたらサーバーレス関数や別サービスへの移行を検討した方がよい。

共有アプリでは、ゲスト利用とログイン利用を同時に残すと責務が広がりやすい。ローカル保存、Supabase同期、プロフィール同期、共有メンバー判定が一箇所に集まると、将来の変更で壊れやすい。リモート同期層とローカル永続化層はさらに分ける余地がある。

秘密情報対策では、削除したつもりでも履歴に残る問題が大きい。公開済みの値は、履歴から消した後でも使われた可能性を前提にローテーションする必要がある。

次に検討した方が良いこと

kenRp-blog では、週次記事とworklog下書きの見出しルールを検証スクリプトへ寄せたい。特にworklogを公開一覧へ出さない方針は、frontmatterだけでなくテストでも守れるようにしておくと安心だ。

shared-shopping-app では、所有者、共有メンバー、未参加ユーザー、公開リンク閲覧者の4パターンでAPIテストを追加したい。Route Handlerごとに重複しているトークン解決や権限判定も、共通ヘルパーへ寄せると保守しやすくなる。

go-sample-api では、CIに gitleaks を組み込みたい。ローカルで一度検査するだけでなく、push前後の自動検査にすると、同じ種類の事故を減らせる。
