Codexの作業ログから公開用の開発記事を作る仕組み

2026年05月04日
#codex#automation#knowledge

Codexとの開発作業は、チャット上では流れが追いやすい。一方で、あとからブログ記事にしようとすると、何を調べ、どのコマンドを実行し、どの判断で実装を変えたのかが散らばりやすい。

そこで今回は、Codexの作業ログとGit履歴から、日次ワークログと週次ブログ記事を生成する worklog-blogger という自作Codexスキルを作った。kenRp-blog はMarkdownファイルを記事として管理している個人技術ブログで、このスキルはそこへ投稿前の下書きを保存する役割を持つ。

行ったこと

最初に、Codexスキルの作り方を確認し、~/.codex/skills/worklog-blogger にスキル本体を作成した。あわせて、スキルをGitHubで管理するために codex-skills リポジトリを用意し、ローカルのスキル配置先へ同期する install-local.sh と、既存スキルをリポジトリへ取り込む sync-from-local.sh を追加した。

実装では、collect_worklog.py にCodexセッションログ、セッションインデックス、Git履歴を読む処理を追加した。さらに、テストを先に書いて、frontmatter、出力パス、Git履歴抽出、ワークログ参照の添付が期待通りになることを確認した。

その後、生成物の方針を見直した。日次でブログ記事まで作ると内容が薄くなるため、日次は詳細なワークログ、週次は他者が読んで学べるナレッジ記事に分ける構成へ変更した。

過程

調査では、まずCodexのスキル作成ガイドと既存スキルの構成を確認した。次に、kenRp-blog の記事形式を確認し、frontmatterに template、path、date、title、thumbnail、metaDescription などが必要なことを見た。

実装時には、以下のようなコマンドを実行して、既存構成、テスト、出力内容を確認した。

sed -n '1,220p' ~/.codex/skills/.system/skill-creator/SKILL.md
find ~/.codex/sessions -path '*2026/05/03*' -name '*.jsonl'
python3 -m unittest discover -s skills/worklog-blogger/scripts -p 'test_*.py'
python3 skills/worklog-blogger/scripts/collect_worklog.py --date 2026-05-03 --mode worklog --dry-run
npm test
git status --short
git add ... && git commit ... && git push

重要な改修箇所は、スキル本文の SKILL.md、Codex UI向け説明の agents/openai.yaml、ログ収集スクリプトの scripts/collect_worklog.py、テストの scripts/test_collect_worklog.py である。GitHub管理用には、README.md、.gitignore、scripts/install-local.sh、scripts/sync-from-local.sh も追加した。

利用した技術

  • Python: collect_worklog.py の実装言語として使った。JSONLの読み取り、Git履歴の収集、Markdown文字列の生成、秘密情報らしい値のマスクを標準ライブラリ中心で実装した。
  • unittest: ログ抽出、Git履歴抽出、frontmatter生成、週次ブログの参照ワークログ添付をTDDで確認するために使った。
  • Markdown frontmatter: kenRp-blog へ保存する記事下書きのメタデータ形式として使った。タイトル、パス、カテゴリ、タグ、下書き状態を記事ファイル内で管理できる。
  • Shell script: install-local.sh と sync-from-local.sh で、Git管理下のスキルと ~/.codex/skills の同期を行うために使った。
  • GitHub private repository: 自作Codexスキルを複数環境で再利用できる正本として管理するために使った。
  • Codex Automation: 実装した worklog-blogger スキルを、日次ワークログ生成と週次ブログ生成の定期実行に接続するために使った。

アーキテクチャ

今回実装した worklog-blogger スキルは、入力、変換、出力、同期の4層に分けた。

入力層では、Codexセッションログ、セッションインデックス、Git履歴を読む。ここは記事本文で説明する「利用技術」ではなく、スキル実装におけるデータソースである。セッションログからはユーザー依頼、Codexの応答、exec_command の実行、apply_patch の変更内容を取り出し、Git履歴からはコードとして残った成果を確認する。

変換層では、collect_worklog.py がログを公開可能な粒度へ整える。秘密情報らしい値は伏せ、内部プロンプトやシステム指示は除外する。今回の改修では、短い要約だけでなく、実際に実行したコマンド、変更ファイル、技術選定や方針判断に関する発話も抽出対象にした。

出力層では、kenRp-blog/_data/blog にMarkdown下書きを保存する。日次ワークログは作業再開のための詳細記録、週次ブログは読者向けの学びとして役割を分けた。

同期層では、codex-skills リポジトリを正本にし、install-local.sh で ~/.codex/skills へ反映する。これにより、普段の編集はGit管理下で行い、Codexが使うローカルスキルへ明示的に同期できる。

選定理由

Pythonを選んだ理由は、今回の実装がローカルファイル、JSONL、日時、正規表現、subprocessを扱う小さなCLIだったためである。外部ライブラリを増やさず、標準ライブラリだけでログ収集からMarkdown生成まで組めるので、Codexスキルに同梱しやすい。

unittestを選んだ理由は、スクリプトがPython単体で完結しており、ログ抽出やfrontmatter生成のような純粋関数に近い処理を軽く検証できるためである。追加のテストフレームワークを入れずにTDDを回せる。

Codexセッションログを入力データとして使う理由は、Git履歴だけでは判断の流れが残らないためである。コミットメッセージからは「何を変えたか」は分かるが、「なぜその構成にしたか」「どの課題で方針を変えたか」は拾いにくい。セッションログを合わせることで、技術選定時の検討や作業中の詰まりどころを記事に反映できる。

GitHubでスキルを管理する理由は、~/.codex/skills を直接編集するだけだと再現性が弱いためである。private repositoryを正本にしておけば、別PCでもcloneして install-local.sh を実行するだけで復元できる。

Markdownを出力形式にした理由は、kenRp-blog の既存構造と相性が良く、生成後に人間がレビューしやすいためである。自動投稿ではなく下書き保存にしているのも、公開前にセンシティブな情報や説明不足を確認するためである。

導入手順

  1. worklog-blogger スキル用のディレクトリを作り、SKILL.md に用途、入力元、出力先、生成ルールを定義する。
  2. test_collect_worklog.py に、ログ抽出、Git履歴抽出、frontmatter生成、週次ブログ生成のテストを書く。
  3. collect_worklog.py に、Codexセッションログ、セッションインデックス、Git履歴を読む処理を実装する。
  4. セッション内の exec_command から実行コマンド、apply_patch から変更ファイル、会話本文から技術選定や判断理由を抽出する。
  5. kenRp-blog のfrontmatterに合わせて、日次ワークログと週次ブログ記事をMarkdownとして出力する。
  6. codex-skills リポジトリを作成し、スキル本体を skills/worklog-blogger 配下へ移す。
  7. install-local.sh でGit管理下のスキルを ~/.codex/skills へ同期できるようにする。
  8. Codex Automationで、日次ワークログ生成と週次ブログ生成を別々の定期実行にする。

導入時の課題

最初の課題は、生成される文章が薄くなりやすいことだった。セッションの要約とGitコミットだけを並べると、読者には「何を実装したのか」「なぜその設計にしたのか」が伝わりにくい。そこで、実行コマンド、変更ファイル、技術選定メモを抽出するようにした。

次の課題は、公開できる情報と内部情報の切り分けである。Codexセッションログには、ローカルパス、認証状態、環境情報、内部指示が混ざることがある。記事では、センシティブな値を伏せ、必要な実装意図だけを残す必要がある。

また、Codexセッションログは別チャットの作業も同じ日付のログとして残るため、対象日の全作業を拾える一方で、記事テーマに関係ない作業も混ざる。今後は、対象リポジトリやスキル名で絞り込む機能が必要になる。

次に検討した方が良いこと

  • exec_command の実行結果から、成功した検証コマンドと失敗した検証コマンドを分けて出す。
  • apply_patch の差分から、変更ファイルだけでなく「何を変えたか」を短く要約する。
  • 対象リポジトリやキーワードでセッションを絞り込み、記事テーマに関係ない作業を除外する。
  • 週次ブログでは、ワークログ本文から技術選定の判断をさらに強く抽出する。
  • 生成後に秘密情報やローカル情報を検出するチェックをテスト化する。
